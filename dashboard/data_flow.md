# Data Flow Diagrams

PlaceIQ depends on seamless transition states bridging human input, the Node.js API, the Agentic pipeline, and the persistent Database.

## 1. The Student Flow (Resume & Recommendation)

This represents the primary asynchronous user journey:

1. **Upload Trigger**: Student uploads a formatted resume on the `/profile` page.
2. **Ingestion**: 
   - Frontend `api.js` pushes the file or text blob via REST.
   - The Backend API verifies the Auth token, parses the request, and triggers the `Orchestrator` agent.
3. **Analysis Phase**:
   - Orchestrator signals the `Resume Analyzer`. It returns a standardized structured `StudentJSON`.
   - The Orchestrator forwards this to the `Readiness Engine`.
4. **Scoring & Matrix Update**:
   - `Readiness Engine` calculates the base factor (e.g. 78/100 readiness).
   - Orchestrator requests data from the `Memory Layer` to gather past contexts.
   - Orchestrator pushes data to the `Company Matcher` and `Selection Predictor`, mapping the `StudentJSON` against all Mock Company profiles.
5. **Decision Synthesis**:
   - The `Decision Engine` consumes the predictor outputs and applies the weighted algorithm (`0.5*Match + 0.3*Prob + 0.2*Time`).
   - The `Action Planner` processes all weaknesses to generate the next actionable step.
6. **Return**: 
   - Orchestrator finalizes the payload, issues a write to the Database (updating the Memory Layer), and completes the HTTP request.
   - Frontend updates Zustand state, refreshing the dashboard visually.

## 2. The Admin Flow (Global Intelligence & Alerts)

The Admin journey heavily relies on aggregated data and real-time alerts rather than single-source interaction.

1. **Dashboard Load**: 
   - Admin accesses the dashboard. API calls fetch metrics generated by the `Admin Intelligence Layer`.
   - Fetches the "At-Risk Dataset" pre-calculated based on students in the bottom 20% of `Readiness Scores`.
2. **Company Injection**:
   - Admin simulates a new company pipeline arriving dynamically. Admin posts new Company requirements via REST API.
3. **Global Re-Calculation (Event Engine Trigger)**:
   - Backend ingests new requirements and triggers a background batch heuristic run.
   - **Condition Met**: If a student's `Decision Engine` score suddenly spikes for this new company, the `Event & Alert Engine` catches the state change.
4. **WebSocket Push**:
   - The `Event Engine` bypasses the REST loop and pushes a `new_high_match` payload directly into the Redis Pub/Sub pool.
   - The Socket.IO server pushes to specific Student Client IDs.
5. **Reaction**:
   - The Student's frontend receives the socket event, fires a Toast notification ("New High Match Found!").
